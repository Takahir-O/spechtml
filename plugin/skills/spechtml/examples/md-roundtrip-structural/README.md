# Markdown round-trip — structural changes (golden samples)

This directory complements `examples/md-roundtrip/` by covering **structural** edits, where the Markdown change implies a non-trivial TOON patch op.

All three scenarios end with `diff after-edit.md rendered-after-patch.md` reporting **zero differences** (verified during the v0.3.2 release session).

## Scenarios

| Dir | Scenario | TOON op used | Verified diff |
|---|---|---|---|
| `01-row-add/` | A new requirement row appended to a `reqs` table in the Markdown view | `append` on `/main/reqs` | 0 |
| `02-row-remove/` | An existing requirement row removed from a `reqs` table | `remove` on `/main/reqs/id=R2` | 0 |
| `03-section-add/` | A brand-new top-level section added to the Markdown view | `add_section` with `key`, `index`, `body` | 0 |

## File layout per scenario

```
<scenario>/
├── source-before.toon     ← starting TOON
├── before-edit.md         ← cli-md output of source-before (what reviewer sees)
├── after-edit.md          ← Markdown after the team-member edit (== rendered-after-patch.md)
├── patch.toon             ← TOON patch generated by the LLM after reading the MD diff
├── source-after.toon      ← TOON after applying the patch (== source-before patched)
└── rendered-after-patch.md ← cli-md output of source-after (verifies the round-trip)
```

`source-after.toon` is generated by the applier in-place, starting from a copy of `source-before.toon`. The reproducible procedure for each scenario is:

```bash
cp <scenario>/source-before.toon <scenario>/source-after.toon

# (optional) recompute target_hash if you regenerate the source:
# node -e "import('node:crypto').then(c => import('node:fs').then(f => process.stdout.write('sha256:' + c.createHash('sha256').update(f.readFileSync('<scenario>/source-after.toon')).digest('hex'))))"

node ${CLAUDE_SKILL_DIR}/runtime/dist/apply-toon-patch.bundle.js <scenario>/source-after.toon <scenario>/patch.toon
node ${CLAUDE_SKILL_DIR}/runtime/dist/cli-md.bundle.js <scenario>/source-after.toon <scenario>/rendered-after-patch.md
diff <scenario>/after-edit.md <scenario>/rendered-after-patch.md   # exit 0
```

## How the LLM picks the op

This mirrors the `Markdown round-trip — strict procedure` section of `SKILL.md`:

1. Run `${CLAUDE_SKILL_DIR}/scripts/md-diff-helper.mjs <before-edit.md> <after-edit.md>` to extract the changed lines + their context.
2. From the context (table header? section header? prose paragraph?), choose:
   - new table row → `append` (or `insert` for ordered tables)
   - removed row → `remove` with `id=...` selector
   - new top-level section → `add_section`
   - prose change → `replace` on the section's `prose`
3. Compute `target_hash`, write the patch, apply, re-render, diff.

Each scenario in this directory is an executable example of one of those branches.
